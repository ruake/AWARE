## 1. Product Overview

### 1.1 Problem Statement

Akamai CDN properties are tested across 3 environment tiers (QA, UAT, PROD) with 2 networks each (staging, production) — 6 environments total. Teams run Playwright browser tests and pytest HTTP/functional tests against these environments on recurring schedules. There is no unified observability layer to:

- Track pass/fail trends across environments over time
- Compare baseline vs candidate runs for regression detection
- Compute flakiness scores and detect anomalous pass-rate drops
- Gate promotions from UAT → PROD based on objective pass-rate thresholds
- Auto-discover test definitions from source code and correlate them with CI results
- Provide an LLM-powered assistant that can answer questions about test health

### 1.2 Solution

A single-page dashboard application that:

- Fetches CI run data, test results, and scheduler state at runtime from a GitHub `data` branch
- Provides a read-only observability layer (no write-back from the app to the data branch)
- Computes derived analytics (pass-rate trends, flakiness, anomaly scores, diff rows)
- Integrates with an AI Copilot for natural-language querying of test health
- Is driven entirely by config-as-code YAML files that define environments, suites, and Akamai properties

### 1.3 Scope

**In scope:**
- Data ingestion from GitHub raw storage (JSON files on `data` branch)
- In-memory reactive stores fed by runtime `fetch()` calls
- Run history browsing, filtering, sorting, and drill-down
- A/B run comparison with computed diff rows
- Pass-rate trend charts, category heatmaps, flakiness leaderboards
- Anomaly detection (Z-score, IQR) at test level and run level
- AI Copilot with LangGraph analysis pipeline and multi-provider LLM support
- Promotion gate tracking (UAT → PROD pass-rate threshold)
- CI pipeline status display
- Test case discovery (pytest AST, Playwright spec parser, Puppeteer, HTTP)
- Test suite hierarchy management
- Environment configuration (6 canonical envs with localStorage override)
- CSV export of run data
- Deep-link sharing via URL search params
- Config-as-code YAML validation against TypeScript types
- Scheduler controller (K8s-style reconciler) for cron-driven workflow dispatch
- Badge generation for latest pass rate

**Out of scope (current):**
- Writing test results back to GitHub from the dashboard (data is CI-written only)
- Real-time WebSocket streaming (data is polled on load and refreshed manually)
- User authentication / authorization (single-user tool)
- Database storage (all data is file-based JSON on orphan branch)
- Mobile native apps
- Multi-tenant / multi-account support

## 3. User Personas

### 3.1 CDN Engineer (Primary)

**Role**: Akamai property owner / CDN reliability engineer  
**Goal**: Monitor test health across 6 environments, detect regressions, gate promotions  
**Needs**:
- See pass-rate trends for each environment over time
- Compare two runs to identify new failures
- Know when UAT pass rate drops below 95% (promotion blocked)
- Investigate per-test failure evidence (HTTP waterfall, assertions)
- Export run data for external reporting
- Quick access to CI pipeline status

**Frequency**: Daily (continuous monitoring) + during deployment windows  
**Power user**: Uses URL deep-links to share specific runs or comparisons

### 3.2 QA Engineer

**Role**: Test author / suite maintainer  
**Goal**: Ensure test coverage is accurate, tests are not flaky, suites are well-organized  
**Needs**:
- Browse auto-discovered tests and verify they map correctly
- Review flakiness scores to identify problematic tests
- Explore test suite hierarchy and suite→test assignments
- Understand which environments each suite targets
- Use AI Copilot to summarize failure patterns

**Frequency**: Per sprint / on test changes

### 3.3 CI/CD Engineer

**Role**: Pipeline maintainer / DevOps  
**Goal**: Ensure scheduler runs correctly, config is valid, runs are recorded  
**Needs**:
- Validate YAML config files against TypeScript types
- View scheduler status (last run, dispatch log, GC activity)
- See CI pipeline state (which workflows are running)
- Generate CI config YAML from the dashboard
- Monitor the reconciler loop

**Frequency**: During pipeline setup/maintenance

### 3.4 Engineering Manager

**Role**: Stakeholder  
**Goal**: High-level view of test health, release confidence  
**Needs**:
- Dashboard KPIs (overall pass rate, trend, regressions)
- Promotion gate status
- About page with system stats and data health
- AI Copilot summarization for standup reporting

**Frequency**: Weekly / per release cycle

## 4. User Journeys

### Journey J-01: Daily Health Check

**Actor**: CDN Engineer  
**Trigger**: Start of shift / before deployment  
**Flow**:
1. Land on Dashboard → see 4 KPI cards (Pass Rate, Total Runs, Failures, Regressions)
2. Inspect 3 TierCards (QA, UAT, PROD) — each shows avg pass rate + env list
3. If any tier shows degradation (critical status), click "Investigate" in AnomalyBanner
4. AnomalyBanner navigates to Compare page with the degraded run as candidate
5. Select a known-good baseline run → view diff rows (regressions in red)
6. Click a regression to see test details → expand assertion evidence
7. Copy the full diagnostic link to share with the team

**Edge cases**:
- AnomalyBanner shown when regressions=0 but data is stale (>24h) → false alert
- Compare page loads with one or both run IDs missing from data → recovery: show dropdown with all runs
- URL params contain invalid run IDs → component should fall back to defaults gracefully

**Success criteria**:
- Engineer identifies failing tests within 3 clicks of landing
- Regression diffs are computed correctly (true positives > false positives)
- No data staleness beyond last CI run

### Journey J-02: Run Investigation

**Actor**: CDN or QA Engineer  
**Trigger**: CI run shows unexpected failures  
**Flow**:
1. Navigate to Runs page (/runs)
2. Filter by environment (e.g., QA), status (FAIL), or search by run ID
3. Click a run row → RunDetail page (/runs/:runId)
4. See summary stats (total/passed/failed/skipped)
5. Filter test results by status: show only failures
6. Expand a failed test row → view assertion details + evidence (HTTP request/response, timings)
7. Optionally click "View Analytics" to see trend for that test over time
8. Click "Compare" to diff this run against a previous passing run

**Edge cases**:
- Run has zero test results → show empty state with helpful message (not "0 results" table)
- TestResult missing evidence field (schema violation) → graceful fallback with "Evidence not available"
- Evidence filmstrip data URIs exceed memory → lazy-load on expand
- runId in URL not found in store → show error state with link to /runs

**Success criteria**:
- Engineer finds root-cause evidence (specific assertion failure) within 5 clicks
- URL params preserve state for sharing the exact view
- Evidence includes complete HTTP waterfall (DNS, TCP, TLS, TTFB, download)

### Journey J-03: Run Comparison (A/B Diff)

**Actor**: CDN Engineer  
**Trigger**: Regression detected / pre-deployment validation  
**Flow**:
1. Navigate to Compare page (/compare)
2. Baseline dropdown auto-selects most recent run; candidate selects the earliest
3. Two run selectors show run ID, env, status, date, pass rate
4. Click Swap button to exchange baseline ↔ candidate
5. View diff row list: each row shows test name, baseline status, candidate status, duration delta
6. Rows color-coded: red = regression, green = fixed, grey = unchanged
7. Search diffs by test name (URL param `q`)
8. Filter diffs by state (regression/fixed only)
9. Click a diff row to drill into that test's details

**Edge cases**:
- Both dropdowns selecting the same run → "Baseline and candidate must be different runs" message
- Baseline date > candidate date → show waring icon but allow (the swap exists for this reason)
- Run with 0 test results produces 0 diffs → empty state
- 500+ diffs → performance degradation (O(n) comparison)
- Duplicate test IDs across runs → deduplication required

**Success criteria**:
- Swap operation is O(1) — no double-inversion bug (confirmed fixed: swapped state deleted)
- Durations delta correctly computed: candidate - baseline
- Regressions and fixes are accurately computed from TestResult[] comparison
- URL params `?baseline=X&candidate=Y` are shareable

### Journey J-04: Flaky Test Investigation

**Actor**: QA Engineer  
**Trigger**: Periodic test maintenance  
**Flow**:
1. Navigate to TestAnalytics page (/trends)
2. View CategoryHeatmap — per-category pass rates across environments
3. View Flakiness leaderboard (tests sorted by flakiness score)
4. Click a flaky test → navigate to its trend / detail
5. Review historical pass/fail pattern to determine if test needs fixing or quarantining

**Edge cases**:
- Zero runs for a category → show N/A instead of 0% (misleading)
- Test with only 1 run → flakiness undefined (needs minimum window of 7)
- All tests pass in window → "No flaky tests detected" (current state — acceptable for this case)
- Category combination exists for env but no runs → empty cell

**Success criteria**:
- Flakiness score is computed from N≥7 run history (alternating pass/fail = high flakiness)
- Category heatmap is data-driven (not hardcoded mock data — currently broken)
- Engineer can identify which tests need attention

### Journey J-05: AI Copilot Analysis

**Actor**: All personas  
**Trigger**: Need for natural-language analysis  
**Flow**:
1. Navigate to Copilot page (/copilot)
2. Choose from 4 suggestion buttons (e.g., "Analyze recent failures", "Compare environments")
3. Type a natural-language query: "Which tests are most flaky in PROD?"
4. Copilot processes query through LangGraph pipeline
5. Response includes rich data (chart, table, etc.) and text summary
6. User can continue the conversation, ask follow-ups
7. Chat history persists to localStorage (7-day TTL)
8. User can start a new thread, bookmark conversations, or use templates

**Critical note**: This journey represents the INTENDED behavior. The CURRENT implementation returns a mock setTimeout response — neither the ai/ nor the copilot/ LangGraph pipeline is wired to the UI. This is the single biggest gap in the application.

**Edge cases**:
- Chrome AI (Gemini Nano) unavailable → show provider configuration prompt
- Query returns no data → "No data matches your query" with suggestion to broaden
- localStorage chat history exceeds 5MB → oldest entries evicted
- 7-day TTL declared but never enforced in code → chat data persists indefinitely
- Rate limiter never called in production pipeline → unlimited requests
- Concurrent queries → abort previous in-flight request

**Success criteria**:
- Real LangGraph pipeline wired to Copilot UI (currently mock-only)
- All data queries go through DataQueries.ts (the only sanctioned data access for AI)
- Responses include at least one chart/table for data-rich queries
- Responses ≤3 sentences of prose (enforced by chart standards)

### Journey J-06: Promotion Gate Check

**Actor**: CDN Engineer / Engineering Manager  
**Trigger**: Before PROD property activation  
**Flow**:
1. Navigate to Dashboard → check PropertyStatusBar (always visible)
2. See current pass rates for UAT/staging and UAT/production
3. If UAT pass rate ≥ 95%, promotion gate is green
4. If below threshold, see which suites are failing
5. Navigate to Settings → Promotion Gates section
6. View/adjust the pass-rate threshold slider (50–100%)
7. Review promotion decision history from data store
8. Make informed decision about PROD activation

**Success criteria**:
- PropertyStatusBar is rendered on every page (app-level)
- 95% threshold is the default; override persists in localStorage
- Promotion history is loaded from promotions.json

### Journey J-07: CI Pipeline Monitoring

**Actor**: CI/CD Engineer  
**Trigger**: During deployment / incident response  
**Flow**:
1. Navigate to Status page (/status)
2. View CI pipeline status: running/completed/failed workflows
3. See scheduler status: last reconcile, dispatch log, GC runs
4. Download or preview CI config YAML
5. Copy config or regenerate as needed

**Success criteria**:
- CI pipeline status reflects real GitHub Actions workflow state
- Scheduler status shows last 50 dispatch entries
- CI config YAML is generated from current store data

### Journey J-08: Test Discovery & Management

**Actor**: QA Engineer  
**Trigger**: After adding new tests to the codebase  
**Flow**:
1. Run `pnpm discover:tests` (or it runs in CI)
2. discover-all.mjs orchestrates 4 discovery scripts
3. Merge strategy: existing filmstrip/screenshot config preserved by scriptPath match
4. Output: data/auto-tests.json
5. In the dashboard, navigate to Tests page (/tests)
6. Browse auto-discovered tests, search by name/ID/description
7. View test details: category, priority, tags, script path
8. Cross-reference with test suites

**Success criteria**:
- Discovery completes in < 30s
- All existing test metadata preserved across re-discovery
- No duplicate test IDs
- Suite→test references remain valid

### Journey J-09: Data Validation & Build

**Actor**: CI/CD Engineer / Developer  
**Trigger**: Before deployment / commit  
**Flow**:
1. Pre-commit hook runs `pnpm verify`
2. Prebuild step (`pnpm build`) runs:
   - `validate-data.mjs`: validates all 7 JSON data files against schema
   - `validate-config.mjs`: validates YAML configs + cross-references + TypeScript alignment
3. If any validation fails → build fails → deployment blocked

**Critical gap**: `validate-data.mjs` loads `test-results.schema.json` but NEVER performs validation against it. The schema file acts as dead weight. Schema validation is implemented as manual TypeScript-level field checks, not against a formal JSON schema.

**Success criteria**:
- Schema validation actually executes against loaded schemas
- Config validation catches: environment/enum drift between YAML and TypeScript
- Build is blocked on any validation error

### Journey J-10: Data Ingestion Lifecycle

**Actor**: System (automated)  
**Trigger**: App mount / page load  
**Flow**:
1. App.tsx mounts → DataGate component rendered
2. DataGate calls `loadAllData()` once (guarded by `triggeredRef`)
3. Phase 1 (parallel): Load runs + test suites
4. Phase 2 (parallel): Load results + promotions + scheduler status + auto-discovered tests
5. Each data module uses `fetchJson<T>(path)` with retry
6. Each module stores data in module-level `let` bindings with pub/sub
7. Components subscribe via `useSyncExternalStore` or custom subscriptions

**Performance concern**: `loadAllData()` triggers 11 `notify()` calls during init — every subscriber recomputes 11 times. Each notify() clones the full data array for `_snapshot`. At 10K+ runs, this causes multi-second freezes on every load.

**Edge cases**:
- Network failure → retry 3× → show error panel with Retry button
- Data loaded but has schema violations → show warning + render what's valid
- Partial failure (some data loaded, some failed) → show warning + render available data
- `_loaded` flag does NOT guard against re-fetch; a second page visit re-fetches all data
- raw.githubusercontent.com has no SLA → rate limiting or 429 responses possible
- `_loaded = true` moved AFTER `await fetchJson()` as retry-block fix, but retry never resets `_loaded` on success

## 5. Data Contracts & Schema

### 5.1 Entity: Run

```typescript
interface Run {
  id: string;
  label: string;
  suiteId: string;
  envId: string;
  env: "QA" | "UAT" | "PROD";
  network: "staging" | "production";
  status: "PASS" | "FAIL" | "RUNNING" | "PENDING" | "ERROR";
  passPct: number;
  failures: number;
  duration: string;
  durationMs: number;
  started: string;
  build: string;
  rev: string;
  conditions?: RunCondition[];
  workflowRunId?: number;
  updatedAt?: string;
}
```

**Conditions** (K8s-inspired status sub-resource):

```typescript
interface RunCondition {
  type: "Dispatched" | "WorkflowRunning" | "Completed" | "Passed" | "Reconciled";
  status: "True" | "False" | "Unknown";
  reason: string;
  message: string;
  lastTransitionTime: string;
}
```

**Status derivation** (from conditions):

| Completed | Passed | Status |
|-----------|--------|--------|
| True | True | PASS |
| True | False | FAIL |
| — | — | RUNNING (if WorkflowRunning=True) |
| — | — | PENDING (dispatched but not active) |
| GC'd | — | ERROR (>24h stale) |

### 5.2 Entity: TestResult

```typescript
interface TestResult {
  id: string;
  testCaseId: string;
  runId: string;
  name: string;
  status: "PASS" | "FAIL" | "SKIPPED" | "FLAKY" | "TIMEOUT" | "ERROR";
  duration: number;
  category: string;
  assertions?: Assertion[];
  tags?: string[];
  error?: string;
  flaky?: boolean;
  screenshots?: string[];
  filmstrip?: FilmstripFrame[];
  retryCount?: number;
  started?: string;
}

interface Assertion {
  assertion: string;
  expected: string;
  actual: string;
  passed: boolean;
}

interface Evidence {
  request: { method: string; url: string; headers: Record<string, string>; body?: string };
  response: { status: number; headers: Record<string, string>; body?: string; timings: Timing };
  assertions: Assertion[];
  filmstrip?: FilmstripFrame[];
}

interface Timing {
  dnsLookup: number;
  tcpConnect: number;
  tlsHandshake: number;
  ttfb: number;
  download: number;
  total: number;
}

interface FilmstripFrame {
  id: string;
  label: string;
  dataUri: string;
}
```

### 5.3 Entity: TestCase

```typescript
interface TestCase {
  id: string;         // ad_0, pw_1, pu_2, ht_3, tc_4
  name: string;
  description: string;
  category: string;
```

## 6. Feature Specifications (by Module)

### 6.1 Data Ingestion (lib/dataFetcher.ts + lib/initData.ts + per-file store modules)

**Purpose**: Fetch JSON data files from GitHub raw storage at runtime; hydrate in-memory stores.

**Key behaviors**:
- `fetchJson<T>(path)` resolves URL based on environment
- Timeout: 10s per request with 3 retries, 1s delay
- Progress tracking: `DataInitState` with 6 boolean readiness flags
- React consumption: `useSyncExternalStore` for each data slice

**Known issues**:
- `_loaded` flag never reset on failure — page reload needed to retry
- 11 notify() cascade during init causes thundering herd
- Raw.githubusercontent.com has no SLA — no fallback/cache mechanism if GitHub is unreachable
- No polling/refresh — data is loaded once per session

### 6.2 Run Management (pages/Runs.tsx + pages/RunDetail.tsx + lib/runs.ts)

**Purpose**: Browse, filter, sort, and inspect CI run records.

**Run History (F-01)**:
- Filter: environment (multi-select), status, suite, text search
- Sort: run ID, env, status, pass rate, duration, started
- Pagination: 30 rows per page (hardcoded)
- CSV export with safe leading-character escaping

**Run Detail (F-02)**:
- Summary: total/passed/failed/skipped counts
- Filter: status, category, text search
- Expandable rows with assertion detail + evidence
- Actions per test: "View Analytics", "View Definition", "Compare"

**Known issues**:
- `computeTestDetailForName` iterates ALL runs × ALL results = O(6000+) per call
- `slice(0,50)` hardcoded pagination — at ~60 runs total, 10 extra rows hidden
- Missing loading state for test result expansion (evidence fetch is sync but large)
- Duplicate test IDs across runs not handled

### 6.3 Run Comparison (pages/Compare.tsx + lib/runs.ts computeDiffRows)

**Purpose**: Compare two runs to detect regressions and fixes.

**Specification**:
- URL: `/compare` with params `baseline`, `candidate`, `q`, `filter`
- Diff computation: index candidate by testCaseId, compare each baseline result
- Status mapping: PASS→FAIL = regression, FAIL→PASS = fixed, same = unchanged

**Known issues**:
- `window.location.href` replaced with wouter `navigate()` (confirmed fix)
- Swap double-inversion removed (`swapped` state deleted, URL swap alone is correct)
- 500+ diffs cause O(n) performance degradation
- No masking/pagination for large diff sets

### 6.4 Dashboard Analytics (pages/Dashboard.tsx + lib/hooks/useData.ts)

**Purpose**: At-a-glance health summary with KPIs, tier health, pass-rate trends.

**KPIs**: Total runs (filtered), overall pass rate, pass trend (delta), failed runs, regression count

**Pass-Rate Trend (F-05)**: Area chart from data-driven pipeline — works correctly.

**Anomaly Banner**: Shown when regressions > 0 or any tier degraded.

### 6.11 Scheduler Controller (scripts/scheduler.mjs)

**⚠️ CRITICAL**: No file-locking mechanism. Two concurrent scheduler instances simultaneously read `runs.json`, modify in memory, and write — the last writer wins, silently discarding the other's changes.

**5 Phases per Tick**: Reconcile → Cron Eval + Dispatch → GC → Persist → Commit

**Known issues**:
- `git checkout data` before writing destroys local working tree if data branch has diverged
- No lockfile or transaction — concurrent instances corrupt data
- No tests (zero coverage for scheduler logic)
- Force-push to data branch with potential history loss
- After checkout, `git add -A` includes stale files from previous checkout

### 6.12 Run Recording (scripts/record-run.mjs)

**Purpose**: Record CI test results from GitHub Actions workflows.

**Known issues**:
- No idempotency: if called twice for same run, creates duplicate entries
- Runs.json matches by suite+envId on RUNNING entry — if no RUNNING entry found, silently prepends new entry (masking misconfiguration)
- No validation that env variables are set before processing

### 6.13 Config Validation (scripts/validate-config.mjs)

**Purpose**: Cross-validate YAML config files against each other and against TypeScript types.

**Known issues**:
- No validation that env IDs in YAML match TypeScript `Run.env` enum
- No validation against actual GitHub workflow names
- Config file size constraints exceed GitHub's 32KB limit in 2 sections (environments.yml expanded IP lists)

### 6.14 Data Validation (scripts/validate-data.mjs)

**⚠️ CRITICAL BUG**: This script loads `test-results.schema.json` into memory but NEVER validates any data against it. The schema file is dead weight. JSON files are validated via manual TypeScript-level field checks, not against formal JSON Schema definitions.

### 6.15 Badge Generation (scripts/generate-badge.mjs)

**Purpose**: Generate SVG health badge for README.

## 7. Workflow Engine

### 7.1 CI/CD Pipeline Flow

```
[External Trigger]        [Cron: 15min]
        │                       │
        ▼                       ▼
   run-tests.yml          scheduler.mjs
  (GitHub Actions)        ├── Reconcile (poll GH)
   ┌──────────────┐       ├── Cron Eval → Dispatch
   │               │      ├── GC (>24h stale)
   └──────────────┘       ├── Persist status
        │                 └── Commit [skip ci]
        ▼                        │
   record-run.mjs                │
   ├── Update run                │
   ├── Enrich evidence            │
   └── Write data files          │
        │                        │
        ▼                        ▼
   ───────────────────────────────────
   data branch (orphan, root level)
   ├── runs.json
   ├── test-results.json
   ├── scheduler-status.json
   └── ...
              │ (runtime fetch)
              ▼
   AWARE Dashboard (React SPA)
   ├── loadAllData() → hydrate stores
   └── Render via useSyncExternalStore
```

**Failure modes**:
- Scheduler and CI both write to data branch simultaneously → last-writer-wins corruption
- `git checkout data` fails if data branch diverged → script exits without recovery
- GitHub API rate limiting (5000 req/hr) → reconcile phase misses updates
- Force-push loses history on data branch

### 7.2 Pre-commit / Build Gate

```
Commit → simple-git-hooks → pnpm verify (tsc + eslint + prettier + vitest)
    │
    ▼ (pass/fail)
CI Push → deploy.yml → validate-data + validate-config + typecheck + test + build
```

### 7.3 Data Flow (Runtime) — Currently Wired

```
raw.githubusercontent.com → fetchJson<T> → loadAllData() → module stores → React hooks → components
```

### 7.4 AI Copilot Pipeline — INTENDED (Currently Not Wired)

```
User query → Copilot page → runGraphAgent() → LangGraph DAG (7 stages) → Response
```

### 7.5 Config Validation Pipeline

```
config/*.yml → validate-config.mjs (4 passes + cross-refs + TypeScript alignment) → exit 0/1
```

## 14. Data Evolution & Versioning Strategy

### 14.1 Current State

- 0 of 7 JSON data files have version fields
- 11 localStorage keys with mixed `_v1`/`_v2`/`_v3`/`_v4`/`_v5` naming (inconsistent: some use underscore, some use hyphen, some suffix-only)
- ZERO migration logic for old localStorage formats
- ZERO migration logic for old JSON data formats
- Schema files exist for `test-results.json` but are never used for validation

### 14.2 Versioning Schema (Required)

Every JSON data file must include:

```typescript
{
  "$schema": "./schemas/runs.schema.json",
  "version": 1,
  "data": Run[]
  // ... actual data
}
```

Files to version:
- `runs.json` → `{ version: 1, data: Run[] }`
- `test-results.json` → `{ version: 1, data: Record<runId, TestResult[]> }`
- `auto-tests.json` → `{ version: 1, data: TestCase[] }`
- `test-suites.json` → `{ version: 1, data: TestSuite[] }`
- `promotions.json` → `{ version: 1, data: PromotionDecision[] }`
- `scheduler-status.json` → `{ version: 1, data: SchedulerStatus }`
- `diff-results.json` → `{ version: 1, data: DiffRow[] }`

### 14.3 Migration Strategy

- JSON Schema per file: `data/schemas/<file>.schema.json`
- Migration script: `scripts/migrate-data.mjs` — reads all files, checks version, applies transforms sequentially
- Backward compatibility: store readers handle missing `version` field as `version: 0` (legacy format)
- localStorage migration: version key embedded in stored JSON, migration function registered in map

### 14.4 localStorage Naming Convention

Standardize to: `aware_<domain>_v<version>` (all lowercase, underscore-separated, consistent `_v` suffix)

## 17. State Management Architecture

### 17.1 Current Patterns Found

| Pattern | Count | Files | Issues |
|---------|-------|-------|--------|
| Module-level `let` + pub/sub | 7 | runs.ts, testSuites.ts, promotions.ts, etc. | — |
| `useSyncExternalStore` wrapper | ~6 | hooks/useTestData.ts, etc. | Shallow clone bug |
| React Query (`useQuery`) | ~5 | hooks/useData.ts | Stale cache by default |
| `useSyncedUrlState` | ~10 | urlState.ts | Race condition during async |
| Event bus (dead letter) | 1 | eventBus.ts | Zero subscribers |
| State machine (generic) | 1 | stateMachine.ts | Orphaned — no consumers |

### 17.2 Known Issues

| Issue | Location | Description | Priority |
|-------|----------|-------------|----------|
| Stale snapshot references | runs.ts: `_runsSnapshot = [..._runs]` | Listeners receive old array until next update if DATA is mutated in place | P1 |
| Zombie observation windows | All stores via `_runs.length = 0; _runs.push(...data)` | Components may hold stale snapshots while DATA is modified | P1 |
| Boolean guards never reset | All stores: `_loaded` | Once true, never allows retry (even if data is stale) | P1 |
| 3 subscription mechanisms | runs.ts: `_runsNotify`, `runsSubscriptionSet`, legacy `set` | Triple redundancy, synchronization not guaranteed | P2 |
| React Query + pub/sub overlap | hooks/useData.ts + stores | Some data loaded twice (React Query cache + module store) | P2 |
| Event bus emits to dead letter | eventBus.ts | 12 events defined, 3 emitted, ZERO subscribers | P2 |

### 17.3 Recommended Architecture

Consolidate to a single pattern:

```
Data files → fetchJson → module stores (single pub/sub) → useSyncExternalStore → React components
```

- Remove event bus (dead code)
- Remove React Query for data that loads at init (keep for lazy loads only)
- Consolidate 3 subscription mechanisms to 1
- Replace `[...DATA]` shallow clone with `structuredClone()` or immutable update pattern
- Reset `_loaded` to false on 3 consecutive failures
- Batch `notify()` calls during `loadAllData()` to prevent thundering herd

## 18. Security Disclosures & Remediation

### 18.1 🔴 CRITICAL: Exposed GitHub OAuth Token

**Finding**: GitHub OAuth token `REDACTED_OAUTH_TOKEN` exposed in plaintext at `.config/gh/hosts.yml`.

**Risk**: Any user with filesystem access can read this token and gain full GitHub API access as user `ruake`.

**Remediation**:
1. Immediately rotate the token via GitHub Settings → Developer settings → Personal access tokens
2. Add `.config/gh/hosts.yml` to `.gitignore` (it is a local auth file, not a project file)
3. Check git history for accidental commits of this file
4. Consider using GitHub CLI's built-in auth mechanism (`gh auth login`) instead of manual token storage

### 18.2 🟡 HIGH: VITE_ Env Vars Inlined in Client Bundle

**Finding**: All `VITE_`-prefixed environment variables are inlined into the client-side JavaScript bundle by Vite at build time.

**Risk**: Any API keys or secrets prefixed with `VITE_` are visible to all users via browser DevTools.

**Remediation**:
- No `VITE_` variable should contain secrets
- OpenAI API key must flow through a server-side proxy only (never to the client)
- Document which variables are intentionally public

### 18.3 🟡 MEDIUM: CSP 'unsafe-inline' Present

**Finding**: Content Security Policy includes `'unsafe-inline'` for script execution.

**Risk**: Reduces effectiveness of XSS protection.

**Remediation**: Remove `'unsafe-inline'` by migrating inline event handlers to addEventListener patterns and using nonce/hash-based CSP.

### 18.4 🟢 LOW: AI Proxy Plugin Accepts Unsanitized Content

**Finding**: The Vite plugin for AI proxy passes user content directly to the LLM API without server-side content filtering or PII scrubbing.

**Risk**: User may inadvertently send PII to external LLM API.

**Remediation**: Add server-side content sanitization in the AI proxy plugin for user queries before forwarding.

### 18.5 🟢 LOW: copyToClipboard API Without Permission Check

**Finding**: Clipboard write operations use `navigator.clipboard.writeText()` without checking `permissions.query({ name: 'clipboard-write' })`.

**Risk**: In some browser contexts, the API call silently fails.

**Remediation**: Add permissions check or use fallback for unsupported contexts.

## 27. Flakiness Computation Specification

### 27.1 Current State — 6 Inconsistent Implementations

| # | Location | Algorithm | Edge: runs=1 | Notes |
|---|----------|-----------|-------------|-------|
| 1 | lib/runs.ts (computeFlakiness) | flips / (runs-1) | Returns 0 | Primary |
| 2 | lib/flakiness.ts (computeFlakinessScores) | flips / (runs-1) | Returns NaN | Secondary |
| 3 | lib/flakiness.ts (calculateFlakinessForTest) | flips / (runs-1) | Returns 0 | Tertiary |
| 4 | lib/ai/useCases.ts (flaky-detection) | flips / (runs-1) | Returns 0 | AI pipeline |
| 5 | lib/copilot/useCases.ts (flaky-detection) | flips / (runs-1) | Returns 0 | Copilot pipeline |
| 6 | lib/anomalyDetection.ts (detectFlakiness) | flips / (runs-1) | Returns NaN | Anomaly system |

**All 6 differ in edge handling when runs < 2, but use the same core algorithm.**

### 27.2 Required Single Specification

```typescript
function computeFlakiness(results: TestResult[]): number {
  // Minimum window: 7 runs
  if (results.length < 7) return undefined;

  // A flip is when status alternates between PASS and non-PASS
  // between consecutive runs
  let flips = 0;
  for (let i = 1; i < results.length; i++) {
    const prev = results[i-1].status === "PASS";
    const curr = results[i].status === "PASS";
    if (prev !== curr) flips++;
  }

  return flips / (results.length - 1); // 0 = stable, 1 = alternates every run
}
```

**Rules**:
- Only one implementation — remove the other 5
- Minimum window: N≥7 (configurable, default 7)
- A "flip" is status alternation between PASS and any non-PASS (FAIL, ERROR, TIMEOUT, FLAKY)
- SKIPPED does NOT count as non-PASS (skip is neutral)
- Result: 0.0 (completely stable) to 1.0 (alternates every run)
- Threshold: >0.3 = flaky, >0.6 = very flaky
- FLAKY status in a single run (from retry/computation) is treated as non-PASS for flip detection

### 28.1 Current State — 4 Non-Integrated Systems

### 28.2 Required Single Architecture

Consolidate to a single unified system with composable strategies:

```typescript
interface AnomalyDetector {
  type: "zscore" | "iqr" | "threshold";
  detect(runs: Run[], results: TestResult[][]): AnomalyReport[];
}

interface AnomalyReport {
  runId: string;
  testId?: string;        // undefined for run-level
  metric: "passRate" | "duration" | "failureCount" | "flakiness";

}
```

**Strategies**:
1. **Z-Score**: 7-day rolling window, threshold |Z| > 1.5 = anomaly
2. **IQR**: Interquartile range, factor 1.5 = outlier
3. **Threshold**: Hard thresholds (pass rate < 95% = warning, < 80% = critical)
